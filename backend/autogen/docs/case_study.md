# 案例整理（Case Study Notes）

本文件用于把项目沉淀成可用于面试、课程展示、论文初稿或研究计划的素材。它不替代 `dev_journal.md` 的实时记录，也不替代 `project_context.md` 的完整上下文。

## 1. 一句话介绍

本项目基于 AutoGen 多智能体框架和 persona 配置，自动生成教育协作场景中的模拟群聊数据，为后续基于 LLM multi-agent 的学生表现评估提供数据来源和工程基础。

## 2. 背景问题

主项目希望评估学生在协作学习或团队任务中的表现，例如参与度、贡献度、协调能力、冲突处理能力和负面行为倾向。

当前难点是缺少可直接使用的数据集。真实课堂协作数据通常存在以下问题：

- 获取成本高。
- 隐私和伦理限制强。
- 标注成本高。
- 不容易控制不同协作行为的比例。
- 难以构造对抗测试场景。

因此当前分支先做“可控模拟数据生成”（controllable synthetic dialogue generation），用 persona 角色和多智能体群聊生成不同类型的小组协作对话。

## 3. 解决思路

项目采用配置驱动的 multi-agent 数据生成方法：

- 用 `persona.json` 定义不同协作人格。
- 用 `configs/experiment_groups.json` 组合不同对照组。
- 用 AutoGen `GroupChat` 模拟多角色互动。
- 用 OpenRouter 调用可用大模型。
- 用增量保存机制保留生成结果。
- 用项目记录体系持续沉淀开发过程和技术决策。

这种方法的核心价值是：在没有真实数据集的前期，可以先构建一个可控、可重复、可扩展的数据生成环境，用于验证后续评估指标和算法设计。

## 4. 当前工程实现

当前已经完成的工程能力：

- 从单个 `persona.json` 扩展成完整 Python 项目。
- 支持动态读取 11 个 persona。
- 支持配置 4-5 人实验组。
- 支持多个对照组，例如正向协作组、混合负面角色组、高冲突组。
- 接入 OpenRouter，并验证可用模型 `qwen/qwen-2.5-7b-instruct`。
- 固定 AutoGen 0.2，使用经典 `GroupChat` 运行线。
- 支持 `--dry-run` 做配置检查。
- 支持按 group 单独运行。
- 输出 `run.log`、`metadata.json`、`state.json`、`dialogues.jsonl` 和 `final_dataset.json`。
- 增量写入 `dialogues.jsonl`，降低中断损失。
- 建立项目记录体系，支持后续复盘和跨项目复用。

## 5. 技术挑战与解决方案

### 挑战 1：AutoGen 版本不兼容

问题：

`autogen-agentchat~=0.2` 实际可能安装到 0.7.x，导致经典 `GroupChat` API 不兼容。

解决：

将依赖固定为：

```text
autogen-agentchat>=0.2,<0.3
```

可表达亮点：

这个问题体现了 Python 依赖版本约束的重要性，也说明工程项目中不能只看包名，必须验证实际安装版本和 API 兼容性。

### 挑战 2：OpenRouter 模型路由不可用

问题：

使用 `openai/gpt-4o-mini` 时出现 403，错误并非 API key 无效，而是模型路由或服务策略不可用。

解决：

先验证 key 和余额，再用最小化请求测试模型，最后切换到可用模型 `qwen/qwen-2.5-7b-instruct`。

可表达亮点：

排查时区分了“认证失败”“余额问题”“模型路由问题”和“项目代码问题”，避免在错误方向上修改代码。

### 挑战 3：对话生成过程可能被中断

问题：

如果只在全部运行结束后保存结果，用户手动终止或系统异常会导致前面生成的内容丢失。

解决：

将输出改为增量保存：

- 每完成一段对话就追加到 `dialogues.jsonl`。
- 同步更新 `state.json`。
- 最后再生成 `final_dataset.json`。

可表达亮点：

这是典型的数据生成任务容错设计（fault tolerance），能显著降低长任务失败成本。

### 挑战 4：多智能体对话难以自然收束

问题：

多个 agent 会持续讨论、重复观点或偏离任务，不能稳定自然结束。

解决：

当前使用两层机制：

- prompt 中要求任务完成时输出 `[GROUP_COMPLETE]`。
- 配置 `max_round` 作为硬上限。

后续改进：

- 增加总结代理（summary agent）。
- 增加裁判代理（judge agent）。
- 增加质量检测器判断是否跑题或重复。

可表达亮点：

这个问题体现了 multi-agent 系统中“自治性”和“可控性”的矛盾，需要同时设计行为约束和工程兜底机制。

## 6. 面试表达素材

可以按以下结构讲述项目：

1. 背景：主项目需要评估教育场景中的学生协作表现，但缺少数据集。
2. 目标：先构建一个可控的 multi-agent 对话数据生成器。
3. 方法：用 persona 定义角色，用 AutoGen GroupChat 模拟小组讨论，用配置文件定义对照组。
4. 工程：实现 CLI、配置加载、群聊运行、增量保存、运行日志和元数据追踪。
5. 难点：AutoGen 版本兼容、OpenRouter 模型路由、对话收束、中断容错。
6. 结果：已经可以生成多组模拟对话，并保存为可用于后续评估的数据文件。
7. 后续：增加弱标注、总结代理、裁判代理和数据质量评估。

## 7. 论文或研究表达素材

可能的研究叙事：

在教育协作评估中，真实学生对话数据稀缺且标注成本高。本文尝试构建一种基于 persona-driven multi-agent simulation 的合成对话生成流程，通过控制小组成员的协作性格组合，生成不同协作质量和冲突水平的群聊数据，为后续学生表现评估模型提供测试数据和对抗样本。

可以进一步展开的研究问题：

- 不同 persona 组合是否能稳定诱发不同协作模式？
- LLM 生成的协作对话能否用于训练或测试学生表现评估模型？
- 如何从群聊中自动提取个体贡献度、领导力、协作维护行为和负面行为？
- 总结代理或裁判代理能否提升生成数据的任务完成度和可标注性？
- 合成数据和真实教育数据之间的差距如何评估？

## 8. 当前证据和不足

当前已有证据：

- 项目可以真实调用模型并生成对话。
- 输出文件可以保存运行结果和元数据。
- 不同 group 可以通过配置区分。
- 中断容错已经通过增量保存机制改善。

当前不足：

- 还没有系统评估生成对话质量。
- 还没有自动标签。
- 还没有和真实学生数据对比。
- 还没有明确的教育任务场景库。
- 还没有形成可发表实验结果。

## 9. 后续阶段目标

下一阶段可优先做：

- 设计教育任务场景库。
- 增加总结代理或裁判代理。
- 设计弱标注 schema。
- 为每段对话生成成员级指标。
- 建立数据质量检查规则。
- 选取少量样本进行人工评审，验证生成数据是否符合预期。

当这些能力完成后，本项目会更接近一个可用于论文实验或面试展示的完整系统。
